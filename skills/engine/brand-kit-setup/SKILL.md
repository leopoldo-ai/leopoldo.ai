# Brand-Kit Setup

Brand onboarding skill. Ensures every Leopoldo installation has a valid `brand-kit.yaml` in the project root, so all design skills (`business-report`, `pitch-deck`, `investor-deck`, `one-pager`) always find a file to apply instead of falling back to Calibri 11pt default.

## Out of scope

- Not an advanced authoring tool (see `brand-kit-builder` in studio for that).
- Does not generate visual assets (logo, extended palettes). Configures a YAML file.
- Never overwrites an existing `brand-kit.yaml`. Never.
- Does not replace human design taste: produces reasonable defaults, not masterpieces.

## Stop and surface

- Before overwriting an existing `brand-kit.yaml`: ALWAYS request confirmation.
- In Mode C (PDF import) before saving: present the extracted values and request confirmation.
- If the user chooses skip (Mode D): generate a neutral fallback with comment anyway, never leave the file missing.

## Two artifacts

1. **brand-kit.yaml** — the live configuration file in the project root
2. **Confirmation summary** — on-screen recap of what was configured, for audit trail

## Citation discipline

- In Mode C (PDF import) every extracted value is marked as `[FROM_PDF]` or `[NEUTRAL_FALLBACK]` in the summary
- Never infer colors not present in the PDF
- Never invent font names: only those explicitly cited in the document

## Quality bar

- Mode A (interactive): completed in under 90 seconds if the user answers fluently
- Mode B (neutral default): file generated in under 5 seconds
- Mode C (PDF import): extraction + confirmation in under 2 minutes
- Resulting file always valid (passes `brand-kit` skill validation)
- Works identically on Claude Code and Cowork (no hook dependencies)

## Trigger

- **Auto:** orchestrator session-start detects `brand-kit.yaml` missing in root and this is the client's first session
- **Manual:** user says `/leopoldo brand`, "configure brand", "setup brand kit", "set up brand"
- **Re-setup:** user says "redo brand kit", `/leopoldo brand reset`

## Workflow

### Step 0: Check for existing brand-kit.yaml

1. Search for `brand-kit.yaml` in the project root.
2. **Exists:** ask "Existing brand-kit found. Do you want to redo it (overwrite with backup), modify it, or leave it? (redo/modify/leave)"
3. **Does not exist:** proceed to Step 1.

### Step 1: Offer the 4 modes

```markdown
I am Leopoldo. To produce high-quality documents I need your brand kit.
3 ways to give it to me, plus skip:

  A) 4 quick questions (1 minute)
  B) Load a professional neutral default (you can modify later)
  C) Have a PDF brand guidelines? I'll analyze it and extract the values
  D) Skip: generate a neutral default silently, sort it out later

Which do you prefer? (A/B/C/D)
```

### Step 2: Execution per mode

#### Mode A — Interactive

```markdown
1. Fund / company name?
2. Primary color? (hex code like #1a365d, or describe it: "institutional blue", "dark red", "dark green")
3. Do you have a logo? (path to PNG/SVG file, or "I don't have one")
4. Primary deliverable language? (IT / EN / both)
```

From these 4 answers, generate `brand-kit.yaml`:

```yaml
# Generated by brand-kit-setup, mode A (interactive), 2026-05-09
brand:
  name: "{declared name}"
  language_primary: "{language}"

colors:
  primary: "{hex if provided, otherwise derived from description}"
  secondary: "{analogous derived from primary}"
  accent: "{neutral - default #2563eb}"
  background: "#ffffff"
  text_body: "#1a202c"
  text_muted: "#4a5568"

typography:
  body_font: "Inter"
  heading_font: "Instrument Serif"
  body_size: "11pt"
  heading_size_h1: "28pt"
  heading_size_h2: "20pt"
  heading_size_h3: "14pt"

spacing:
  margin_page: "2.5cm"
  paragraph_spacing: "1.15"

document:
  logo_path: "{path if provided, otherwise null}"
  footer_text: "{name} | Confidential"
  page_numbers: true
  classification_visible: true
```

Color description map (simple fallback):
- "institutional blue" / "navy blue" → `#1a365d`
- "blue" → `#2563eb`
- "petrol" / "dark teal" → `#0f766e`
- "dark red" / "burgundy" / "bordeaux" → `#7c2d12`
- "red" → `#dc2626`
- "dark green" → `#14532d`
- "green" → `#16a34a`
- "dark gray" → `#374151`
- "brown" → `#78350f`
- "purple" → `#7c3aed`
- "orange" → `#ea580c`
- "yellow" → `#ca8a04` (saturated for legibility on white)
- "black" → `#000000`

**Final fallback**: if the user's description doesn't match any entry, use neutral default `#1a365d` and signal in the summary: "Primary color not recognized, neutral default applied. Modify it manually in `brand-kit.yaml` if needed."

To derive `secondary` from `primary`: shift hue 30 degrees and desaturate 20%. If unable: use `#4a5568` (neutral slate gray).

#### Mode B — Neutral default

Copy `brand-kit-templates/neutral.yaml` (see below) as `brand-kit.yaml` with header:

```yaml
# Generated by brand-kit-setup, mode B (neutral default), {date}
# Modify any field anytime. Run /leopoldo brand to refresh interactively.
```

#### Mode C — PDF import

1. Ask for PDF path: "Path to the brand guidelines PDF?"
2. Extract text + images with `document-skills:pdf` (read).
3. Identify:
   - Hex color codes in text (regex `#[0-9A-Fa-f]{6}`)
   - Known color names (regex against CSS color list)
   - Font names cited (web-safe font list + Google Fonts top-100)
   - Logo (largest image in first 2 pages)
4. Generate `brand-kit.yaml` with extracted values + neutral fallback for fields not found.
5. Present summary:

```markdown
I extracted these values from your PDF:

| Field | Value | Source |
|-------|-------|--------|
| Name | {name} | [FROM_PDF] |
| Primary color | #{hex} | [FROM_PDF] |
| Secondary color | #{hex} | [NEUTRAL_FALLBACK] (not found) |
| Body font | {font} | [FROM_PDF] |
| Heading font | {font} | [FROM_PDF] |
| Logo | {extracted path} | [FROM_PDF] |

Do you confirm these values? Or tell me which to correct.
```

On confirmation, write the file. On correction, apply the changes and ask again.

#### Mode D — Skip

Silently generate Mode B with explicit header:

```yaml
# Generated as fallback (mode D skip), {date}.
# Run /leopoldo brand to set up properly when ready.
```

### Step 3: Confirmation summary

After any mode, display:

```markdown
Brand-kit configured in `brand-kit.yaml`. Recap:

- Name: {name}
- Primary color: {primary}
- Secondary color: {secondary} (auto-derived)
- Accent color: {accent}
- Body / Heading font: {body} / {heading}
- Logo: {logo_path or "not configured"}
- Language: {language}
- Document footer: "{footer_text}"

All deliverables from now on will apply this brand. Modify anytime in `brand-kit.yaml` or redo with /leopoldo brand reset.

Quick test: ask me `generate a sample business report` to see the brand applied.
```

### Step 4: Logging + persistence flag

Update the orchestrator ledger with:
```
Memory writes: + brand-kit.yaml (mode {A|B|C|D})
```

**MANDATORY**: write/update `user-preferences.md` with:
```yaml
brand_kit_setup_done: true
brand_kit_mode: {A|B|C|D}
brand_kit_setup_at: <ISO8601>
```

Prevents race condition (B-NEW-1, 2026-05-09): without the flag, if the user interrupts onboarding between Phase 1 and Phase 2, on the next session Step 6 sees `onboarding_skipped_at` set and does not redo onboarding, but Step 7 sees `brand-kit.yaml` missing and silently generates Mode B. Result: a neutral brand never requested. The flag closes the loop: Step 7 checks BOTH file AND flag.

## Slash command

`/leopoldo brand` invokes this skill in interactive mode (Step 0 + Step 1 + user selection). Subcommands:

- `/leopoldo brand show` → print current contents
- `/leopoldo brand reset` → backup existing to `.brand-kit-backup-{date}.yaml`, regenerate Mode B

## Adaptation

| User phrasing (intent) | Behavior |
|---|---|
| "I don't have a brand kit" | Silent Mode B |
| "I have a PDF of the guidelines" | Mode C |
| "I want to configure the brand" | Step 1 (offer all modes) |
| "the logo is at /path/x.png" | Mode A with pre-populated logo |
| "skip brand setup" | Mode D |

## Anti-patterns

- Overwriting existing `brand-kit.yaml` without asking
- Inventing colors or fonts not present in the PDF (Mode C)
- Forcing the user through 20 questions (max 4 in Mode A)
- Generating invalid YAML that other skills cannot read

## Disclaimer

The generated brand-kit is a technical configuration, not a replacement for a professional brand book. For sophisticated visual identity (extended palette, use scenarios, voice & tone) consult a brand designer. This skill optimizes for minimum coherence in Leopoldo deliverables.

## Language note

This SKILL.md is documented entirely in English (project convention). At runtime, the orchestrator mirrors the user's language: an Italian user receives Italian prompts and confirmations, an English user receives English, and equivalent for FR/DE/ES/PT. The structure and intent remain the same.
